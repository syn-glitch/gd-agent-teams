# 교육용 챗봇 — 토큰 관리 및 사용 트래킹 시스템

> 개발팀 회의용 초안 | 2026-03-19 | 벙커 팀 작성

---

## 1. 배경 및 목적

### 1.1 현재 상태

- 학생 대상 코딩 실습 + 자연어 기반 AI 질의응답 챗봇 운영 중
- **기능 1**: 코딩 실습 (파이썬 문제은행, 마이크로비트, 라즈베리파이)
- **기능 2**: 교과목 질의응답 (정보, 기술 분야 — 인공지능, 데이터분석 등)

### 1.2 해결하려는 문제

| 문제 | 설명 |
|------|------|
| 비용 통제 부재 | 학생별 사용량 제한 없이 무제한 사용 → 비용 예측 불가 |
| 사용 패턴 미파악 | 학생이 코딩 실습/교과 질문 중 어디에 집중하는지 데이터 없음 |
| 학습 분석 불가 | 학교 유형별·학생별 활동 비교 분석 불가 |

### 1.3 목표

1. 학생 1인당 **한화 1,000원** 상당의 토큰을 부여하고, 사용 시 차감
2. 학생이 **자기 사용 현황**을 대시보드에서 실시간 확인
3. 관리자가 **학생별·학교별·학교유형별** 사용 데이터를 필터링·분석

## 3. 토큰 경제 시스템 설계

### 3.1 토큰 부여 정책

| 항목 | 내용 |
|------|------|
| 1인당 예산 | 한화 1,000원 |
| 토큰 환산 | API 단가 기준 환산 (예: 1,000원 ≒ 약 XX만 토큰) |
| 부여 시점 | 학생 계정 등록 시 자동 부여 |
| 소진 시 처리 | **(논의 필요)** 사용 중단 / 추가 충전 / 관리자 수동 충전 |

> **개발팀 논의 포인트**: API 모델별(Sonnet/Haiku) 단가가 다르므로, 토큰을 원화 기준(크레딧)으로 관리할지, 실제 토큰 수 기준으로 관리할지 결정 필요

### 3.2 토큰 차감 방식

```
학생 질문 입력
    │
    ▼
┌──────────────────┐
│ 1. 요청 분류       │  ← 기능1(코딩실습) / 기능2(교과질문) 자동 태깅
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 2. API 호출        │  ← 입력 토큰 + 출력 토큰 측정
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 3. 토큰 차감       │  ← 잔여 크레딧에서 실사용량 차감
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 4. 사용 로그 기록   │  ← 트래킹 DB에 상세 기록
└──────────────────┘
```

### 3.3 요청 분류 기준 (자동 태깅)

| 분류 | 카테고리 | 세부 태그 예시 |
|------|----------|---------------|
| 기능 1 | 코딩 실습 | `python`, `microbit`, `raspberry-pi` |
| 기능 2 | 교과 질문 | `ai-concept`, `data-analysis`, `info-tech` |

> **개발팀 논의 포인트**: 자동 분류 방식 — 규칙 기반(키워드) vs LLM 기반(의미 분석) vs 사용자 선택(UI에서 카테고리 선택)

## 4. 사용 트래킹 데이터 모델 (초안)

### 4.1 핵심 엔티티

```
[Student] ─────────── [School]
    │                     │
    │ 1:N                 │ 속성: school_type
    ▼                     │ (마이스터고/자사고/일반고)
[Usage_Log]               │
    │                     │
    │ 속성:               │
    │ - session_id        │
    │ - timestamp         │
    │ - category (기능1/2) │
    │ - sub_tag           │
    │ - input_tokens      │
    │ - output_tokens     │
    │ - cost_krw          │
    │ - query_text (요약)  │
    │ - response_length   │
    ▼
[Token_Balance]
    │
    │ 속성:
    │ - total_granted (부여량)
    │ - total_used (사용량)
    │ - remaining (잔여량)
```

### 4.2 주요 테이블 초안

**students**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| student_id | UUID (PK) | 학생 고유 ID |
| name | VARCHAR | 학생 이름 |
| school_id | FK → schools | 소속 학교 |
| grade | INT | 학년 |
| created_at | TIMESTAMP | 등록일 |

**schools**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| school_id | UUID (PK) | 학교 고유 ID |
| school_name | VARCHAR | 학교명 |
| school_type | ENUM | `meister` / `autonomous_private` / `general` |
| region | VARCHAR | 지역 |

**usage_logs**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| log_id | UUID (PK) | 로그 고유 ID |
| student_id | FK → students | 학생 |
| session_id | UUID | 세션 단위 묶음 |
| timestamp | TIMESTAMP | 요청 시각 |
| category | ENUM | `coding_practice` / `subject_qa` |
| sub_tag | VARCHAR | `python`, `microbit`, `ai-concept` 등 |
| input_tokens | INT | 입력 토큰 수 |
| output_tokens | INT | 출력 토큰 수 |
| cost_krw | DECIMAL | 원화 환산 비용 |

**token_balances**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| student_id | FK → students (PK) | 학생 |
| total_granted | DECIMAL | 부여 금액 (원) |
| total_used | DECIMAL | 사용 금액 (원) |
| remaining | DECIMAL | 잔여 금액 (원) |
| last_updated | TIMESTAMP | 최종 갱신 시각 |

## 7. 개발팀 논의 포인트

### 7.1 기술 의사결정 필요 사항

| # | 논의 주제 | 선택지 | 권장안 |
|---|----------|--------|--------|
| 1 | **크레딧 단위** | (A) 원화 기준 관리 (B) 토큰 수 기준 관리 | (A) 원화 — 모델 변경에도 일관된 예산 관리 가능 |
| 2 | **요청 자동 분류** | (A) 규칙 기반(키워드) (B) LLM 분류 (C) UI 선택 | (C) UI 선택 + (A) 키워드 보조 — 분류 비용 최소화 |
| 3 | **토큰 소진 정책** | (A) 사용 차단 (B) 관리자 수동 충전 (C) 자동 알림 후 차단 | (C) — 잔여 20% 시 알림 → 0원 시 차단 → 관리자 충전 가능 |
| 4 | **대시보드 기술 스택** | 기존 챗봇 프론트에 통합 vs 별도 페이지 | 기존 챗봇 내 탭/메뉴로 통합 권장 |
| 5 | **데이터 저장소** | RDB (PostgreSQL) vs NoSQL (MongoDB) | 집계·필터 쿼리 중심이므로 RDB 권장 |
| 6 | **실시간 vs 배치** | 잔액 실시간 반영 vs 5분 단위 집계 | 잔액은 실시간, 대시보드 차트는 5분 캐시 |

### 7.2 단계별 개발 로드맵 (제안)

```
Phase 1 — MVP (핵심 기능)
├── 토큰 차감 엔진 (API 호출 후 사용량 기록)
├── 학생 잔여 크레딧 조회 API
├── 학생 대시보드 (잔여량 + 최근 내역)
└── 예상 기간: ___주

Phase 2 — 트래킹 강화
├── 요청 자동 분류 (카테고리 태깅)
├── 학생 대시보드 차트 (분야별·일별)
├── 관리자 대시보드 MVP (필터 + 테이블)
└── 예상 기간: ___주

Phase 3 — 분석 고도화
├── 학교유형별 비교 분석 차트
├── CSV 다운로드
├── 토큰 추가 부여 기능
├── 사용 패턴 리포트 자동 생성
└── 예상 기간: ___주
```

### 7.3 확인 필요 사항 (기존 시스템)

- [ ] 학생 인증/로그인 체계가 현재 어떻게 구성되어 있는지?

- [ ] 현재 API 호출 로그가 남고 있는지?

## 8. 용어 정리

| 용어 | 설명 |
|------|------|
| 토큰 | LLM API에서 처리하는 텍스트 단위 (입력+출력) |
| 크레딧 | 토큰 사용량을 원화로 환산한 내부 화폐 단위 |
