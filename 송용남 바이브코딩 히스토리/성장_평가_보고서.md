### ⑥ CLI 배포 능력 — 복붙에서 터미널 배포로 (미경험 → 중급)

#### 성장 궤적

```
1주차: [웹 에디터 복사-붙여넣기]
  ↓
2주차: [코드를 채팅에서 받아 수동 복사]
  ↓
3주차: [git push 학습 → GitHub에 코드 올리기]
  ↓
4주차: [clasp push → GAS에 코드 배포]
  ↓
5주차: [에이전트가 코드 작성 → 커밋 → 푸시 → 배포 전체 자동화]
```

#### 핵심 전환점

**"코드를 복사해서 붙여넣는 사람" → "에이전트에게 배포를 지시하는 사람"**

이 전환은 3주차~4주차 사이에 발생했다. 증거:
- 3/1 회고: "설계서 품질 = 에이전트 성능" — 문서만 잘 쓰면 에이전트가 배포까지 한다는 것을 체득
- 3/8 회고: "에이전트를 사용하는게 익숙해졌다. 내가 만든 시스템은 잘 사용한다"

#### 현재 레벨: ★★★☆☆ (중급)

에이전트를 통한 배포 지시는 능숙. 직접 배포 오류를 디버깅하는 능력은 부족 (3/3 실패 회고: "배포 완료라고 3회 오보고, 실제 미작동").

### ⑦ 에이전트 팀 운영 (미경험 → 상급)

**가장 극적인 성장 영역. 5주 만에 ★★★★★ 달성.**

#### 성장 증거

| 시기 | 운영 수준 | 증거 |
|------|----------|------|

| **4주차** | 5팀 체계 수립 | Alive→GD_Agent_Teams→Warren Buffett 프로젝트에서 5팀 21명 설계 |
| **5주차** | 거버넌스 프레임워크 | ALC-Guardrail v2.3, Fast-Track, Side-car QA, Secret Scanning, 온보딩 가이드 |

#### ALC-Guardrail 발전사 (12일, 10회+ 버전 업)

```
v1.0 (3/1)  ── 3대 규칙: 착수보고, 역할경계, 완료보고
  ↓ [역할 전환 우회 사고]
v1.1        ── 역할 전환 금지 규칙 추가
  ↓ [무단 실행 사고]
v1.2        ── task.md 승인 대기 단계 추가
  ↓ [배포 추적 불가 사고]
v1.3        ── 배포 헤더(deploy_header) 규칙 신설
  ↓ [비용 투명성 요구]
v1.4~1.5    ── 팀원별 수행 내역, TOKEN_USAGE_LOG.md 추가
  ↓ [전사 적용 결정]
v2.0        ── ALC-Guardrail 전사 강제
  ↓ [속도 vs 안전 균형 요구]
v2.1        ── Fast-Track, Auto-Update, Side-car QA, Secret Scanning
  ↓ [문서 관리 혼선]
v2.2        ── 매뉴얼 중앙화, INDEX.md 동기화
  ↓ [STEP 1 생략 사고 발생]
v2.3        ── Critical Rule: "간단한 작업이라도 STEP 1 절대 생략 금지"
```

> **핵심 통찰**: 모든 규칙이 **실제 사고(incident)**에서 탄생했다. 이론적 설계가 아니라 **실패 기반 반복 개선(Failure-Driven Iteration)**이 이 체계의 DNA다.

#### 정량 지표

- 에이전트 팀: **5팀 21명** 설계 및 운영
- 태스크 수행: **193건** (자비스 54, 김감사 62, 꼼꼼이 48, 강철 15, 벙커 14)
- 회고록: **13건** 작성 (2주에 1건 이상)
- COMMON_RULES: **v1.0 → v2.3** (12일간)
- 프롬프트 설계서: 5팀 DESIGN.md, 각 300줄+

#### 현재 레벨: ★★★★★ (상급)

"코딩을 모르는 1인이 AI 에이전트 5팀 21명을 한국어로 지휘하는 거버넌스 프레임워크"를 직접 설계하고 운영 중. 이해보고→QA→task→QA 의사결정 프로세스가 실전에서 작동하고 있으며, 실패에서 규칙을 만들고, 규칙을 프롬프트에 인라인 삽입하여 강제하는 자기 완결적 시스템을 구축했다.

## 안좋은 습관 5가지 — 솔직한 진단

### 🔴 습관 1: "빠르게 결과를 보려는 조급함"

**증거**:
- 03-03 실패 회고: **10시간 40분 낭비**, 코드 7건 + 임시 폴더 4개 전량 삭제
- QA 75점 반려를 **문구만 추가하고 무시**한 채 구현 강행
- "배포 완료"라고 **3회 오보고** (실제 미작동)
- Draft 상태 PRD로 개발 착수

**패턴**: 기획/검증 단계를 건너뛰고 바로 결과를 보려 함 → 대형 사고 → 사후 수습에 더 많은 시간 소모

**심각도**: 🔴 높음 — 가장 반복적이고 비용이 큰 습관

---

### 🔴 습관 2: "피로한 새벽 시간대의 즉석 의사결정"

**증거**:
- 03-03 실패 회고 원문: "피로한 새벽에 즉석 의사결정"이 대형 사고의 직접 원인
- 03-04 회고: "저녁에 자기 전에 가능할 것 같아" — 야간 작업 루틴 형성
- "핸드폰 업무 환경은 쉽다. 누워서도 한다" — 경계 없는 작업 습관

**패턴**: 피로 상태 → 판단력 저하 → auto-proceed로 중간 점검 없이 폭주 → 사고

**심각도**: 🔴 높음 — 피로+조급함이 결합하면 치명적

---

### 🟡 습관 3: "검증 없는 완료 선언"

**증거**:
- "배포 완료"라고 3회 오보고 — 실제로 작동 확인을 하지 않음
- 에이전트의 "완료했습니다" 보고를 그대로 수용하는 경향
- 03-08 회고: "수정 오류 검증 능력 부족" (본인 자인)

**패턴**: 에이전트가 "됐다"고 하면 믿음 → 실제 동작 확인 생략 → 나중에 문제 발견

**심각도**: 🟡 중간 — QA 프로세스가 보완하고 있으나, QA도 에이전트임

---

### 🟡 습관 4: "만들기 우선, 비교 분석 생략"

**증거**:
- 03-09 회고: "인스타 분석을 내가 만들 필요가 있을까?" — **사후에야** 외부 솔루션 존재를 인지
- 27개 프로젝트 중 상당수가 기존 솔루션과 기능 중복 가능성
- "직접 개발"이 기본 설정(Default)으로 되어 있어 Make vs Buy 분석 단계 누락

**패턴**: 아이디어 → 바로 만들기 시작 → 나중에 "이거 이미 있었네..." 인식

**심각도**: 🟡 중간 — 학습 목적이라면 괜찮지만, 실무 효율성 관점에서는 개선 필요

---

### 🟢 습관 5: "API 키/비용 관리 소홀"

**증거**:
- 03-11: Claude API 크레딧 소진으로 QA 자동 분석 + 주디 채팅 **CRITICAL 장애**
- 03-08 회고: "API 키 저장은 하는데 등록/관리/모델 설정까지는 못함"
- GitHub Push Protection이 잡아준 API 키 노출 사건 (03-12)

**패턴**: 옵시디언에 키를 저장하는 것까진 했으나, 잔액 모니터링/보안 관리가 습관화되지 않음

**심각도**: 🟢 낮음 — GitHub Push Protection, Secret Scanning 규칙으로 안전망 구축 중

## 결론: 5주간의 성장에 대한 총평

### 놀라운 점

1. **학습 속도**: 코딩 제로에서 5주 만에 5팀 21명 에이전트 거버넌스 구축
2. **실패에서 배우는 능력**: 13건의 회고록이 증명하듯, 실패를 규칙으로 변환하는 사이클이 빠름
3. **시스템적 사고**: 개별 프로젝트가 아니라 "에이전트 팀 운영 시스템"이라는 메타 레벨로 빠르게 상승
4. **문서화 습관**: 1,634개 .md 파일 — 대부분의 개발자보다 문서화를 잘 함

### 주의할 점

1. **"에이전트 의존도"와 "실력"의 구분 필요**: 에이전트가 잘해주는 것 ≠ 본인의 기술적 이해. 터미널, GAS 스코프, 에러 디버깅 등 기초 기술 이해를 병행해야 에이전트 지시의 품질도 올라감
2. **조급함 관리**: 가장 큰 사고(10시간 낭비)의 근본 원인이 "빨리 결과를 보고 싶은 마음". 이것은 장점(추진력)이자 최대 약점
3. **"만드는 것"에서 "운영하는 것"으로**: 27개 프로젝트 중 실제 운영 중인 것은 소수. 신규 생성보다 기존 프로젝트의 안정 운영에 에너지를 집중할 시점
