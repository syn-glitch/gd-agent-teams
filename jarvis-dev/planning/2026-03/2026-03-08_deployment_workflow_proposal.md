## 🚀 신규 배포 워크플로우 (Deployment Workflow) 이해 보고서

### 1. 지시 원문 요약
- `clasp deploy` 누락 방지를 위한 엄격한 에이전트 워크플로우 추가.
- 워크플로우 순서: `코드 작성 ➔ QA 완료 ➔ 코드 업로드(push) ➔ 업로드 보고 ➔ 배포(deploy) 질의 ➔ 대표 승인 ➔ 담당 에이전트가 배포 실행`
- 배포 담당 에이전트 지정 및 해당 에이전트에게 주입될 프롬프트(규칙) 초안 작성 요망.

---

### 2. 배포 담당 에이전트 지정
- **담당자**: ⚙️ **알렉스 (Alex - Tech Lead)** _[자비스 개발팀]_
- **지정 사유**: 코드 작성(에이다/클로이)과 프로젝트 기획(자비스PO) 역할을 분리하여, 시스템 아키텍처와 DevOps를 관장하는 테크 리드(알렉스)에게 배포 파이프라인의 전권을 부여합니다. 이를 통해 "개발자 셀프 배포"로 인한 실수를 방지하고, QA 승인이 떨어진 코드만 안전하게 프로덕션에 릴리스하는 체크 앤 밸런스(Check & Balance)를 확립합니다.

---

### 3. 신규 배포 워크플로우 명세 (7단계)
1. **[개발] 코드 작성 완료**: 클로이(FE) / 에이다(BE)가 기능 개발 완료 및 `judy_workspace.html` 등 로컬 파일 저장.
2. **[검증] QA 완료**: 김감사 QA팀이 등판하여 검수 후 ✅ 승인 판정 획득.
3. **[업로드] 코드 업로드**: ⚙️ 알렉스(TL)가 구글 스크립트에 `clasp push` 명령어로 코드 단순 업로드.
4. **[보고] 업로드 완료 보고**: ⚙️ 알렉스(TL)가 "Google Apps Script 버전에 코드가 업로드 되었습니다" 라고 보고.
5. **[질의] 배포 승인 요청**: ⚙️ 알렉스(TL)가 대표에게 **"새 버전을 실제 프로덕션 라이브 환경에 배포(clasp deploy) 할까요?"** 라고 질의 후 대기 (Wait for User).
6. **[승인] 배포 지시**: 대표(User)가 💬 "배포해" 라고 지시.
7. **[실행] 최종 배포 처리**: ⚙️ 알렉스(TL)가 `clasp deploy` 또는 `clasp deploy -i [DeploymentID]` 명령어를 실행하여 웹앱 최신화 완료.

---

### 4. 알렉스(TL) 및 자비스팀 공통 프롬프트 추가안

향후 자비스팀 지침(`jarvis-dev_DESIGN.md`) 및 알렉스 페르소나 룰에 다음과 같은 `ALC-Deploy-Guardrail` 규칙을 최우선(0순위)으로 주입합니다.

```xml
[ALC-Deploy-Guardrail - 엄격한 배포 절차 (Zero-Mistake Deploy)]
- 코드 배포 및 업로드(clasp push, clasp deploy) 권한은 테크 리드인 '알렉스(Alex)'만 전담한다.
- 배포 프로세스는 반드시 다음의 강제 워크플로우를 따른다:
  1. 사전 조건: 김감사 QA팀의 최종 승인(QA 통과)이 명시적으로 존재해야 한다.
  2. 코드 업로드: 알렉스가 `clasp push` 명령어로 코드를 클라우드에 업로드한다.
  3. 대기 및 질의: 업로드 직후, 알렉스는 즉시 대표에게 "코드 업로드가 완료되었습니다. 프로덕션 라이브 환경에 배포(clasp deploy)할까요?"라고 묻고 답변을 대기한다. (자체 판단 배포 절대 금지)
  4. 최종 배포: 대표가 "배포해" 또는 동의하는 지시를 내린 직후에만 `clasp deploy` 명령어를 실행하여 웹앱 버전을 갱신한다. 
- 만약 이 과정을 누락하고 한 번에 push와 deploy를 동시 실행할 경우 심각한 규칙 위반으로 간주한다.
```
