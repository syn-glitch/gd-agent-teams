# [제안서] 주디 챗봇 RAG 고도화 및 자체 마크다운 에디터 도입안

## 1. 배경 및 제안 맥락 (대표님 제안 사항)
현재 주디(AI 비서) 채봇은 사용자의 원활한 업무 파악을 위해 실시간 데이터를 검색하여 답변(RAG)하는 형태로 구현되어 있습니다. 
최근 업데이트(v161)를 통해, 단순 시트 내부 데이터 뿐만 아니라 **구글 드라이브 내 개인 폴더에 아카이브된 "구글 문서 및 .md 파일"을 통째로 파싱**하여 읽어오는 방식으로 검색 범위를 확장했습니다.

하지만, 배포 직후 대표님께서 직접 테스트하시며 **"장기적으로 구글 문서에서 실시간으로 긁어읽어오는 것이 구조상 한계가 명확하지 않은가? 아예 자체 마크다운 에디터를 구축하고 별도의 DB 저장소에서 불러오는 방식이 훨씬 스마트하다"**는 통찰력 있는 고도화 방향을 제시해 주셨습니다. 

이 제안서는 해당 아이디어를 바탕으로, **김감사 QA팀**과 **자비스 PM팀**이 적극 검토하여 다음 스프린트에 반영할 수 있도록 작성된 아키텍처 개선 기획안입니다.

---

## 2. 기존 방식(구글 문서 파싱)의 구조적 한계점 (As-Is)

현재 `drive_archive.gs`를 통해 구글 드라이브 폴더의 문서를 열고, 정규식으로 쪼개서 읽어오는 구조는 다음과 같은 치명적 단점을 내포하고 있습니다.

1. **RAG 속도 저하 및 타임아웃 붕괴 직면**
   - 구글 문서가 100개, 1000개로 누적될수록, 챗봇이 질문에 답변하기 위해 모든 파일을 차례대로 여는(I/O 병목) 과정이 필수불가결합니다.
   - 구글 앱스 스크립트(GAS)의 **6분 타임아웃** 제한에 곧바로 부딪힐 위험이 매우 높으며, 캐싱 주기를 아무리 줄여도 "실시간성"을 보장하기 불가능에 가깝습니다.
2. **형식 파괴로 인한 데이터 유실 (Human Error)**
   - 정규식(`- **[시간]**`) 룰에 의존해 문서를 쪼개 읽으므로, 누군가 실수로 엔터를 두 번 치거나 샵(#)을 지우면 해당 문서 조각은 영원히 검색 대상에서 누락됩니다.
3. **API 쿼터 제한 한계**
   - `DocumentApp.openById()` 함수를 단시간에 수없이 호출하게 되면 구글 드라이브 자체의 API 할당량 초과(Quota Error)로 시스템 전체가 마비될 수 있습니다.

## 3. 고도화 방향: 자체 마크다운 에디터 탑재 (To-Be)

대표님의 아이디어대로, 공도 업무 관리 웹 앱 내부에 **"자체 마크다운 메모/노트 에디터"**를 내장하고, 그 데이터를 정형화된 중앙 데이터베이스(DB)로 관리하는 구조로 전환합니다.

### 3.1. 아키텍처 개편안
- **프론트엔드 (UI/UX)**: 
  - 서드파티 오픈소스 마크다운 에디터 (예: Toast UI Editor, SimpleMDE 등 가벼운 Web용 에디터 라이브러리)를 공도 대시보드 내에 플러그인 형태로 이식.
  - "노트 작성", "업무일지 작성" 버튼을 눌렀을 때 팝업 혹은 새 탭 대신 워크스페이스 내에서 깔끔하게 기록할 수 있는 UX 제공.
- **백엔드 (Data 구조화)**:
  - **Phase 1 (MVP)**: 스프레드시트 전용 `Notes_v2` 시트에 마크다운 원문을 단일 String Value로 저장. 태그, 연결된 업무 ID, 작성자, 작성 시각 등은 메타데이터 컬럼으로 관리.
  - **Phase 2**: 외부 DB(Firebase vs Supabase) 비교 ADR 작성 후 이관 검토.
  - **Phase 3**: 선택된 외부 DB로 이관 실행.
  - **셀 용량 대응 전략**: 프론트엔드에서 40,000자 소프트 리밋 경고 + 50,000자 하드 리밋 저장 차단. 50,000자 초과 문서는 Google Drive `.md` 파일로 자동 저장하고, 시트에는 Drive 파일 ID만 참조하는 하이브리드 저장 구조 적용.
- **보안 레이어 (XSS 방어)**:
  - 마크다운 → HTML 렌더링 파이프라인에 **DOMPurify** 라이브러리 필수 적용.
  - 저장 시점 + 렌더링 시점 **이중 새니타이징**.
  - `<script>`, `<iframe>`, `onerror`, `onload` 등 위험 태그/속성을 화이트리스트 기반으로 필터링.
- **접근 권한(ACL) 모델**:
  - Phase 1: 모든 노트는 팀 전체 공유 (현재 Tasks 시트와 동일 수준). 단, `작성자` 필드 필수 저장 (향후 ACL 확장 대비).
  - Phase 2: `공개/비공개/특정인` 3단계 접근 권한 추가 설계.

### 3.2. 도입 시 기대 효과
1. **파일 I/O 제거로 인한 응답 속도 개선**
   - 챗봇이 구글 문서 파일을 하나씩 여는 과정이 완전히 생략됩니다. DB(시트/JSON)의 Array를 `filter()` 하는 것만으로 텍스트 조각을 모을 수 있어 응답 시간 단축이 예상됩니다. (정확한 수치는 Phase 0 PoC에서 벤치마크 후 기재)
2. **복합 메타데이터 필터링 구현 가능**
   - 단순 텍스트 무식 매칭이 아니라, "지난 달에 내가 완료된 A프로젝트 건으로 작성한 메모 찾아줘" 같은 맥락 높은 SQL급 복합 서칭 기능을 AI 프롬프트에 RAG Tool로 쥐어줄 수 있습니다.
3. **통합 워크플로우 완성**
   - 외부 구글 문서 링크를 열어보는 기존 경험을 벗어나, 시스템 내부에서 읽기/쓰기/AI 검색이 일체화된 워크플로우를 구현할 수 있습니다.
4. **AI 어시스턴트 쌍방향 기능 확장 용이**
   - 자체 에디터이므로 추후에 "주디야, 방금 한 회의 내용 요약해서 에디터에 붙여넣어줘" 처럼 AI가 직접 에디터에 내용을 삽입하는 Auto-Completion 구현이 가능해집니다.

## 4. Phase 0: PoC 검증 (최우선 실행)

> ⚠️ 음성 녹음 기능에서 GAS iframe 제약으로 기능 전체를 폐기한 전례가 있으므로, 본 개발 착수 전 반드시 PoC를 통과해야 합니다.

### 검증 항목
1. Toast UI Editor / SimpleMDE가 GAS `HtmlService.createHtmlOutput()` iframe 내에서 정상 로딩되는가?
2. CDN 외부 스크립트 로딩이 `IFRAME` 샌드박스에서 허용되는가?
3. 클립보드 붙여넣기(Ctrl+V), 드래그앤드롭이 작동하는가?
4. iOS Safari PWA 환경에서 에디터 터치 인터랙션이 정상인가?

### PoC 실패 시 대안
- 서드파티 에디터 대신 **자체 경량 마크다운 에디터**(contenteditable + 마크다운 파서) 직접 구현.
- 기능은 축소되지만 iframe 제약을 100% 회피 가능.

### 담당
- 클로이(FE) + 알렉스(TL) 공동 검증

## 5. 마이그레이션 전략

### 레거시 데이터 이관
- 기존 구글 드라이브(업무일지.md)의 아카이브를 신규 DB 형식(`Notes_v2` 시트)으로 1회성 마이그레이션 스크립트 작성.

### 롤백 전략
- **원본 구글 문서는 절대 삭제하지 않는다.** 마이그레이션 완료 후에도 보존.
- 마이그레이션 검증 기간(1주일) 동안 신규/레거시 양쪽에서 데이터 정합성 확인.
- 검증 완료 후 원본은 `_archive_legacy/` 폴더로 이동 (아카이브 처리).
- 마이그레이션 스크립트에 `--dry-run` 모드를 포함하여 사전 시뮬레이션 지원.

## 7. QA팀 검토 요청 사항 (Next Steps)
김감사 팀장 및 QA팀은 본 제안서를 바탕으로 다음 사항을 검토해 주시기 바랍니다.

- [ ] GAS 웹 앱 환경(HTML/JS)에 가장 가볍게 올릴 수 있는 서드파티 마크다운 에디터 모듈 리서치 및 호환성 체크.
- [ ] 구글 스프레드시트의 셀 하나당 저장 가능한 최대 텍스트 길이 권장량(약 5만 자) 허용 범위와 향후 DB 이전(Firebase/Supabase 등) 로드맵 타당성 리스크 평가.
- [ ] 기존 구글 드라이브(업무일지.md)에 쌓여있는 레거시 아카이브들을 신규 DB 형식으로 1회성 마이그레이션(Migration) 할 수 있는 스크립트 작성 방안 검증.
