## 1. 배경 및 목적

### 왜 이전하는가?

공도 업무 관리 시스템(Judy Workspace)은 Google Sheets를 DB로 사용하고 있었으나, 다음 문제가 반복 발생:

| 문제 | 현상 |
|------|------|
| 속도 저하 | 업무 목록 로딩 3~5초, 데이터 많아질수록 악화 |
| 전체 스캔 | 매번 시트 전체를 불러와서 코드에서 필터링 (인덱스 없음) |
| 동시 접속 | Lock 기반 동시성 제어 (10초 타임아웃) |
| 스케일 한계 | 1만 행 이상 시 체감 성능 저하 |

### 이전 전략

```
[변경 전] 슬랙 봇 → GAS(서버) → Google Sheets(DB)
[변경 후] 슬랙 봇 → GAS(서버) → Supabase PostgreSQL(DB)
                                ↑ UrlFetchApp으로 REST API 호출
```

- GAS 서버는 그대로 유지 (슬랙 봇, 워크스페이스 UI 변경 없음)
- 프론트엔드(judy_workspace.html) 변경 없음
- 사용자 입장에서 달라지는 건 **속도**뿐

### 2.1 Supabase 프로젝트 생성

| 항목 | 값 |
|------|-----|
| 프로젝트명 | GongDo |
| 플랜 | FREE |
| 리전 | Northeast Asia (Tokyo) |
| 환경 | PRODUCTION / main |
| 프로젝트 URL | `https://blwnxxqgiavyajignqxx.supabase.co` |

### 2.2 API 키 확인

Supabase가 2026년 키 체계를 변경하여 기존 `anon`/`service_role` 대신 새 이름 사용:

| Supabase 새 이름 | 기존 이름 | GAS Script Properties 키 |
|-----------------|----------|--------------------------|
| **Publishable key** (`sb_publishable_...`) | anon key | `SUPABASE_ANON_KEY` |
| **Secret key** (`sb_secret_...`) | service_role key | `SUPABASE_SERVICE_KEY` |

> ⚠️ **중요 발견**: Secret key(구 service_role)는 REST API 직접 호출이 차단됨. Supabase 새 보안 정책에 의해 서버 환경(Node.js 등)에서만 사용 가능. GAS의 UrlFetchApp은 "브라우저"로 판정되어 차단됨. → 모든 API 호출을 **Publishable key(anon) + RLS** 조합으로 전환.

### 2.3 테이블 생성 (SQL 실행)

Supabase SQL Editor에서 `supabase/migrations/20260316_001_initial_schema.sql` 전체를 실행하여 **12개 테이블** 일괄 생성:

```
projects, users, tasks, task_references, notes,
action_logs, calendar_events, token_usage,
agent_tasks, agent_logs, expenses, memo_edit_logs
```

포함 내용:
- CHECK 제약조건 (tasks.status, tasks.priority, projects.status)
- 외래 키 (tasks → projects, task_references → tasks, notes → tasks)
- 인덱스 6개 (tasks 4개 + action_logs 2개)
- RLS 정책 (12개 테이블 모두 anon_full_access)
- 트리거 (updated_at 자동 갱신 3개 + duration_min 자동 계산 1개)

### 2.4 GAS Script Properties 등록

| 속성 | 값 |
|------|-----|
| `SUPABASE_URL` | `https://blwnxxqgiavyajignqxx.supabase.co` |
| `SUPABASE_ANON_KEY` | Publishable key 전체 값 |
| `SUPABASE_SERVICE_KEY` | Secret key 전체 값 (마이그레이션 참고용, 실제 미사용) |

#### 시행착오: Script Properties 50개 초과

GAS Script Properties에 `SESSION_*`, `MAGIC_*` 토큰이 수십 개 누적되어 새 속성 추가 불가.

**해결**: `cleanupOldProperties()` 함수를 작성하여 필수 키(`STORED_SS_ID`, `SUPABASE_*`, `SLACK_*`, `GITHUB_*` 등)만 보존하고 나머지 전부 삭제.

#### 시행착오: SLACK_TOKEN 삭제 사고

`cleanupOldProperties()`의 보존 목록에 `SLACK_BOT_TOKEN`으로 작성했으나, 실제 속성 이름은 `SLACK_TOKEN`이어서 삭제됨. → 슬랙 매직링크 발급 불가.

**해결**: Slack API 페이지에서 Bot User OAuth Token(`xoxb-...`)을 다시 복사하여 `SLACK_TOKEN` 속성으로 재등록.

**해결**: 마이그레이션 코드에서 유효하지 않은 상태값을 자동으로 `"활성"`으로 변환하는 로직 추가.

## 4. Phase 2: tasks 핵심 테이블

**시작**: PM 10:57 | **완료**: PM 11:00 | **소요**: 약 3분 (3번 시도)

### 4.1 첫 번째 시도

```
PM 10:57:33  migratePhase2 실행
PM 10:57:34  프로젝트 매핑: 18개
PM 10:57:34  유효 레코드: 68건 / 건너뜀: 17건
PM 10:57:35  ❌ "duplicate key value violates unique constraint 'tasks_task_id_key'"
```

**원인**: Tasks 시트에 동일한 task_id가 여러 행에 존재 (예: 수정 이력이 별도 행으로 남아있음).
**해결**: INSERT → UPSERT(`ON CONFLICT DO UPDATE`)로 변경. PostgREST의 `Prefer: resolution=merge-duplicates` 헤더 + `on_conflict=task_id` 파라미터 사용.

### 4.2 두 번째 시도

```
PM 11:00:11  migratePhase2 재실행
PM 11:00:15  ❌ "ON CONFLICT DO UPDATE command cannot affect row a second time"
```

**원인**: 같은 배치(100건) 안에 동일 task_id가 2개 이상 포함. PostgreSQL은 한 번의 INSERT 명령에서 같은 행을 두 번 처리할 수 없음.
**해결**: 배치 INSERT 전에 **시트 내 중복 task_id 제거** (마지막 행 우선) 로직 추가. `68건 → 중복 제거 → 20건` 고유 레코드로 축소.

### 4.3 세 번째 시도 → 성공

```
  유효 레코드(중복 전): 68건 / 건너뜀: 17건
  ⚠️ 시트 내 중복 task_id 48건 제거 → 20건
  배치 1: 20건 UPSERT
  ✅ tasks UPSERT 완료: 20건
  [tasks 검증] 시트: 20건 / DB: 20건 → 건수 일치!
  ✅ Phase 2 마이그레이션 완료!
```

### 4.4 tasks 20건 상세 (건너뜀 17건 설명)

- **건너뜀 17건**: Tasks 시트 행 2~18에 업무 ID(A열)와 제목(E열)이 비어있는 빈 행 (드롭다운만 존재)
- **중복 48건**: 동일 task_id가 여러 행에 존재 (시트 내 수정 이력 잔존)
- **최종 20건**: 고유 task_id 기준 마지막 행 데이터로 UPSERT

## 13. 회고: 슈베이스(Supabase DevOps) 에이전트 역할 평가

### 설계서상 역할

자비스 개발팀에는 **🦸 슈베이스 DevOps** 에이전트가 팀원으로 정의되어 있다.
담당: Supabase 환경설정, DB 스키마·마이그레이션·데이터 이전, GitHub Actions CI/CD.

### 실제 기여도: 독자적 기여 없음

이번 마이그레이션에서 슈베이스 에이전트는 **별도로 활동하지 않았다.**

| 작업 | 설계서상 담당 | 실제 수행자 |
|------|------------|-----------|
| DB 스키마 SQL 작성 | 🦸 슈베이스 | 자비스 PO (기존 파일 활용) |
| `supabase_client.gs` 작성 | 🦸 슈베이스 / 👩‍💻 에이다 | 자비스 PO가 에이다 역할 겸임 |
| `supabase_migration.gs` 작성 | 🦸 슈베이스 / 👩‍💻 에이다 | 자비스 PO가 에이다 역할 겸임 |
| Supabase 대시보드 SQL 실행 | 🦸 슈베이스 | **대표님 직접 수행** |
| GAS Script Properties 등록 | 🦸 슈베이스 | **대표님 직접 수행** |
| RLS 정책 확인 | 🦸 슈베이스 | **대표님이 SQL Editor에서 확인** |
| `web_app.gs` 듀얼 모드 전환 | 👩‍💻 에이다 | 자비스 PO가 에이다 역할 겸임 |

### 원인 분석

슈베이스 에이전트가 독자적으로 활동하지 못한 근본 원인:

1. **Supabase 대시보드 접근 불가** — AI 에이전트는 Supabase 웹 콘솔에 로그인하여 SQL 실행, 테이블 확인, 키 조회를 할 수 없다. 결국 대표님이 직접 SQL Editor에서 쿼리를 실행하고 스크린샷으로 결과를 전달하는 방식으로 진행됨.

2. **GAS Script Properties 접근 불가** — API 키 등록, 토큰 관리 등은 GAS 웹 에디터에서만 가능하며 AI 에이전트가 직접 조작할 수 없다.

3. **역할 분리 효과 미미** — 현재 팀 구조에서 "코드 작성"은 한 세션의 AI가 수행하므로, 슈베이스/에이다/알렉스를 별도 에이전트로 분리해도 실제로는 같은 AI가 모든 코드를 작성한다. 역할 분리는 문서상 책임 구분에만 기여.

### 결론

> 현재 구조에서 Supabase 관련 작업은 **자비스 PO + 에이다 BE가 코드 작성**, **대표님이 Supabase/GAS 실행**하는 **2인 체제**로 충분했다. 슈베이스 에이전트의 독자적 기여는 이번에 없었으며, AI가 Supabase에 직접 접근할 수 있는 환경(MCP 서버 등)이 갖춰지기 전까지는 별도 에이전트로서의 실효성이 낮다.
