## 1. 개요
자비스 개발팀이 제안한 '주디 실시간 채팅 기능' 기획안에 대해 개발 착수 전 선제적 품질 검조를 수행함. 본 기능은 사용자의 전방위적 데이터(메모, 업무, 일정)를 통합하여 AI가 답변하는 핵심 기능으로, GAS 환경의 제약 사항을 고려한 설계가 핵심임.

## 2. 영역별 QA 의견 및 리스크 식별

### 🔍 [기능] 다중 데이터 취합 및 성능 리스크 (담당: 테스터)
- **리스크**: `Notes`, `Tasks`, `Calendar` 일정을 매 채팅 시마다 전수 조사할 경우, 데이터 누적에 따라 GAS의 6분 타임아웃 또는 API 응답 지연 발생 가능성 높음.
- **QA 권고**: 
    - **증분 업데이트/캐싱 전략**: 모든 데이터를 매번 긁어오기보다, `CacheService`를 활용하여 5~10분 단위로 데이터를 요약 저장하거나 변경 사항만 반영하는 '컨텍스트 요약' 레이어 도입 필요.
    - **데이터 제한**: 초기 버전에서는 최근 1개월 내 일정이나 활성 메모 위주로 범위를 제한하는 로직 검토 요망.

### 🛡️ [보안] 데이터 접근 권한 및 API 보안 (담당: 보안감사관)
- **리스크**: 실시간 채팅은 웹 앱의 `google.script.run`을 통해 호출되므로, 민감한 개인 데이터(메모 등)가 포함됨. 
- **QA 권고**: 
    - **인증 무결성**: 실행자(`Session.getActiveUser()`)가 시트의 소유자 또는 정당한 열람 권한자인지 API 진입점에서 재검증 필수.
    - **API Key 관리**: `CLAUDE_API_KEY`는 반드시 `PropertiesService`에서 관리하며, 코드 내 노출이나 로그 출력을 엄격히 금지함.

### 🎨 [UX] 사이드바 인터페이스 및 모바일 대응 (담당: UX검증관)
- **리스크**: 우측 사이드바는 가로 폭이 좁아, 긴 채팅 메시지나 표 형태의 데이터 응답 시 가독성이 급격히 저하될 우려가 있음.
- **QA 권고**: 
    - **유연한 레이아웃**: 채팅창 내 메시지는 '말풍선' 형태보다는 Notion 스타일의 블록 형태를 권장하여 가로 공간 활용도를 높일 것.
    - **입력창 고정**: 대화 내용이 길어져도 하단 입력창은 고정(Sticky)되어야 하며, 모바일에서는 가상 키보드가 올라올 때 레이아웃이 깨지지 않도록 `viewport` 대응 필요.

## 3. 통합 QA 판정 및 제언
- **판정**: **조건부 승인 (Proceed with Caution)**
- **제언**: 자비스 개발팀은 구현 계획 수립 시 위 **'캐싱 전략'**과 **'모바일 키보드 대응'**을 핵심 개발 요소에 반드시 포함시키기 바람. 특히 대화 내역의 저장 방식(휘발성 vs 비휘발성)에 대해 대표의 의사를 반영한 명확한 데이터 스키마 설계가 선행되어야 함.
