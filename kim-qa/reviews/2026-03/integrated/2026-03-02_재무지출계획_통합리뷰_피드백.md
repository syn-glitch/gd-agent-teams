## 2. 기능 검토 (🔍 테스터)

### 2.1 기능 누락 검토

| # | 항목 | 판정 | 피드백 |
|---|------|------|--------|
| 1 | 계획 **수정/삭제** 흐름 | 🔴 **차단** | PRD에 등록 흐름만 있고, **수정·삭제 흐름이 완전히 누락**됨. 팀원이 예상 금액을 잘못 입력하거나, 예정일이 변경되는 것은 실무에서 반드시 발생함. 수정 모달/페이지 + 삭제(소프트 삭제 권장) + 이력 추적 기능을 추가 정의해야 함. |
| 2 | **중복 계획 등록** 처리 | 🟡 개선필요 | 기존 캐시플로우 봇에 `checkDuplicate()` 패턴이 있으므로, 동일 항목+유사 금액+유사 일자의 중복 감지를 추가 제안. 차단 사유는 아님. |
| 3 | **1:N 매칭** (카드 2건 → 계획 1건) | 🟡 개선필요 | 교통비 왕복(KTX 편도 2건), 재료 분할 구매 등 1:N 케이스가 실무에서 빈번함. MVP에서는 "추가 매칭" 버튼으로 같은 계획에 여러 건을 연결할 수 있어야 함. PRD에 이 케이스를 추가 정의 필요. |
| 4 | 매칭 **오류 시 해제/재연결** | 🟡 개선필요 | 오매칭 해제 기능이 없으면 데이터 정합성이 깨짐. 웹 페이지에서 "연결 해제" 버튼 + 지출계획 시트 상태 롤백 로직 필요. |
| 5 | 알림 **끄기/빈도 조절** | 🟡 개선필요 | 알림 피로 문제가 PRD 리스크에도 명시됨. 최소한 "이 계획 알림 끄기" 옵션은 MVP에 포함 권장. |
| 6 | 관리자 **승인/반려** 플로우 | ✅ 통과 | 현재 구조에서 관리자는 채널에서 확인만 하는 것이 적절. 승인 플로우를 추가하면 오히려 업무 병목이 발생. **현 설계가 맞음.** |
| 7 | 네이버 API **검색 실패 시 폴백** | 🔴 **차단** | 네이버 API 장애, 타임아웃, API 키 만료, 검색 결과 0건 시 에러 핸들링이 정의되지 않음. GAS에서 `UrlFetchApp.fetch()` 실패 시 사용자에게 어떤 메시지를 보여줄지, 수동 입력으로 폴백할지 정의 필수. |
| 8 | 네이버 결과 + 수동 비교가 **동시 표시** | 🟡 개선필요 | 두 데이터 소스의 표시 순서, 우선순위, UI 구분이 필요. "🤖 자동 검색" vs "✍️ 수동 입력" 라벨로 구분 제안. |

### 2.2 GAS 환경 제약 검토

| # | 항목 | 판정 | 피드백 |
|---|------|------|--------|
| 1 | 일일 알림 트리거 실행 시간 | 🟡 개선필요 | `checkPlanReminders()`가 지출계획 시트 전체를 스캔하는 구조인데, 계획이 100건 이상이 되면 6분 타임아웃 가능. **페이지네이션 또는 배치 처리** 설계를 추가 권장. |

| 3 | 기존 `processAllPendingSaves` + 신규 트리거 충돌 | 🟡 개선필요 | 기존 1분 트리거와 신규 일일 트리거가 동시 실행될 가능성 있음. 트리거 충돌 방지 로직(Lock 기반) 추가 권장. |

## 3. 보안 검토 (🛡️ 보안감사관)

| # | 항목 | 판정 | 피드백 |
|---|------|------|--------|
| 1 | DM 프라이버시 웹 적용 | 🔴 **차단** | 웹 페이지에서 **타 팀원의 지출 계획이 보이면 안 됨**. 기존 Slack DM은 자동으로 개인별 분리가 되지만, 웹 페이지는 명시적으로 **사용자 인증 + 데이터 필터링** 로직이 필요. 현재 `judy_workspace.html`의 인증 방식(접근 제어)을 확인하고 동일하게 적용해야 함. 정의 없으면 개발 착수 불가. |
| 2 | 네이버 API 키 보관 | 🔴 **차단** | PRD에 API 키 관리 방식이 정의되지 않음. **반드시 `PropertiesService.getScriptProperties()`에 저장**하고, 코드에 하드코딩하는 것을 절대 금지. 기존 `Code.gs`의 SLACK_BOT_TOKEN, CLAUDE_API_KEY가 현재 하드코딩 상태인 것도 참고 — 이번 개발에서는 이 패턴을 반복하지 않아야 함. |
| 3 | Claude API 민감 정보 | ✅ 통과 | 차이 분석에 전달되는 데이터(항목명, 금액, 날짜)에 카드번호 등 민감 정보는 포함되지 않음. 현 설계 적절. |
| 4 | PDF 업로드 Drive 접근 권한 | 🟡 개선필요 | 견적서 PDF의 Drive 공유 범위를 명시해야 함. 기존 영수증과 동일하게 `ANYONE_WITH_LINK + VIEW` 권한이면, 링크 유출 시 견적서 열람 가능. 관리자만 접근 가능하도록 제한 권장. |
| 5 | 웹 페이지 인증 방식 | 🔴 **차단** | 웹 페이지 접근 시 **누가 접속했는지 식별하는 방식**이 PRD에 없음. GAS 웹앱의 경우 `Session.getActiveUser().getEmail()`을 사용하거나, Slack OAuth를 통한 인증이 필요. 이것이 정의되지 않으면 프라이버시 구현 자체가 불가능. |

## 4. UX 검토 (🎨 UX검증관)

| # | 항목 | 판정 | 피드백 |
|---|------|------|--------|
| 1 | 네이버 검색 결과 카드 UI | ✅ 통과 | 시나리오에 표시된 카드 UI(이미지+가격+링크+반영 버튼) 구조는 적절. 벨라의 디자인 시안이 필요하지만 UX 방향은 올바름. |
| 2 | 모바일 반응형 | 🟡 개선필요 | 지출 계획 등록 폼의 **모바일 레이아웃**이 정의되지 않음. 네이버 검색 결과 카드가 좁은 화면에서 어떻게 표시되는지 벨라의 모바일 시안 필요. |
| 3 | 웹 + Slack 이중 경로 UX | 🟡 개선필요 | 웹 페이지에서 등록 + Slack `/지출계획`으로도 등록 가능하면, **두 경로의 기능 차이**를 사용자에게 안내해야 함. 예: Slack은 간편 등록(필수 필드만), 웹은 상세 등록(네이버 검색 포함). 이 차이가 PRD에 명시되지 않음. |
| 4 | 분석 대시보드 시각화 | ✅ 통과 | 월말 분석을 웹 대시보드로 시각화하는 방향은 UX적으로 우수. 차트·그래프로 한눈에 파악 가능. |
| 5 | 로딩·에러·빈 상태 UI | 🟡 개선필요 | 네이버 API 검색 중 로딩 스피너, 검색 결과 0건 시 빈 상태 안내, 에러 시 재시도 버튼 등 **3종 상태 UI**를 정의해야 함. |

## 5. 아키텍처 결정 — 김감사 팀장 의견

> 자비스 PO가 질문한 **옵션 A(기존 통합) vs 옵션 B(별도 분리)**에 대한 QA 관점 의견입니다.

### 🏆 김감사 권장: **옵션 B — 별도 분리 (`expense_planner.html`)**

| 판단 근거 | 설명 |
|----------|------|
| **배포 안정성** | 현재 `judy_workspace.html`은 3,871줄이며 주디노트, 칸반, 캘린더 3개 모듈이 작동 중. 여기에 지출관리(등록폼+네이버API+목록+대시보드)를 추가하면 **5,000줄 이상**이 되어, 1줄 수정에도 전체 기능 회귀 테스트가 필요해짐. |
| **QA 테스트 범위** | 통합 시 지출관리 수정 → 칸반·주디노트 회귀 테스트 필수. **분리 시 독립 테스트 가능** → QA 시간 50% 절감. |
| **팀 분업** | 에이다(BE)가 지출관리 백엔드를 수정할 때 `judy_workspace.html`을 건드리면 클로이(FE)의 작업과 **Git 충돌 위험**. 별도 파일이면 병렬 개발 가능. |
| **모바일 PWA** | 분리해도 PWA manifest에 별도 경로를 추가하면 동일한 모바일 앱 경험 제공 가능. |
| **디자인 일관성** | CSS 변수 파일을 공통 모듈로 추출하면 (`common_theme.css`), 별도 파일이어도 동일한 Dark/Light 테마 적용 가능. 코드 중복 우려는 공통 모듈화로 해결. |

### 연결 방안

```
judy_workspace.html                 expense_planner.html
┌────────────────┐                 ┌────────────────┐
│ 📝 주디노트     │                 │ 💰 지출관리     │
│ 📋 칸반보드     │ ──💰 탭 클릭──▶ │  계획 등록       │
│ 📅 캘린더       │                 │  네이버 최저가    │
│ 💰 지출관리 →   │ ◀──🔙 돌아가기── │  분석 대시보드    │
└────────────────┘                 └────────────────┘
       (기존 코드 무변경)               (신규 독립 파일)
```

- `judy_workspace.html`의 GNB에 **💰 지출관리** 탭을 추가하되, 클릭 시 `expense_planner.html`로 **페이지 이동** (iframe 아님)
- 두 파일 모두 동일한 CSS 변수(공통 테마) + GAS 백엔드를 공유
- 기존 워크스페이스 코드는 **탭 링크 1줄 추가만** 수정
