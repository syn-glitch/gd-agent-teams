## 1. iOS Safari 호환성 분석 정확성 — ✅ 정확 (이슈 1건)

| 항목 | 판정 | 소견 |
|------|------|------|
| MediaRecorder iOS 14.5+ | ✅ 정확 | Apple 공식 릴리즈 노트와 일치 |
| getUserMedia iOS 14.3+ | ✅ 정확 | HTTPS 조건 명시 양호 |
| Web Speech API 미지원 | ✅ 정확 | iOS Safari 미지원 사실 정확, Whisper 대안 적절 |
| audio/mp4 포맷 분기 | ✅ 정확 | WebM 불가 → MP4 폴백 전략 적합 |
| **[MINOR-01]** `audio/wav` 폴백 | ⚠️ 검증 필요 | 5.1절 코드에서 `audio/wav` 3순위 폴백인데, MediaRecorder가 WAV를 네이티브 지원하는 브라우저가 거의 없음. 실질적으로 도달 불가능한 코드 경로일 수 있음 → 폴백 시 에러 처리(사용자 안내)로 변경 권장 |

## 2. 리스크 분석 충분성 — ⚠️ 보완 필요 (누락 3건)

**기존 리스크 5건**: 적절히 식별됨. 영향도/발생확률 분류 합리적.

**누락된 리스크**:

| # | 누락 리스크 | 영향도 | 설명 |
|---|-----------|:------:|------|
| **[MAJOR-01]** | **iOS 백그라운드 탭 녹음 중단** | 높음 | 5.2절에서 "경고 표시"만 언급했으나, 실제로 iOS Safari는 탭 전환 시 MediaRecorder 스트림이 **즉시 종료**됨. 녹음 데이터 유실 가능. 대응 전략(자동 저장, 중단 감지 후 부분 저장)이 구체적으로 필요 |
| **[MAJOR-02]** | **PWA(홈 화면 추가) 환경에서의 마이크 권한 지속성** | 중간 | iOS PWA는 세션마다 마이크 권한을 재요청할 수 있음. 현재 주디 워크스페이스가 PWA로 운영되는 만큼, 권한 재요청 UX 플로우가 필요 |
| **[MAJOR-03]** | **Base64 전송 시 메모리 부족 (iOS Safari)** | 중간 | iOS Safari의 메모리 제한이 데스크톱보다 엄격함. 10분 녹음(~10MB) → Base64(~13.3MB) 문자열을 메모리에 올릴 때 저사양 iPhone에서 탭 크래시 가능. 청크 분할 전송 전략 명시 필요 |

## 4. 보안/프라이버시 관점 — ⚠️ 보완 필요 (이슈 3건)

| # | 이슈 | 심각도 | 설명 |
|---|------|:------:|------|
| **[MAJOR-04]** | **음성 데이터 프라이버시 정책 미기재** | MAJOR | 음성은 **생체 정보**에 해당. 녹음 데이터의 보관 기간, 접근 권한, 삭제 정책이 문서에 없음. 특히 외부 API(Whisper)로 음성 전송 시 OpenAI 데이터 처리 정책 확인 및 사용자 고지 필요 |
| **[MAJOR-05]** | **Drive 음성 파일 접근 권한** | MAJOR | Drive 폴더의 공유 설정 명시 없음. 비공개(본인만 접근) 설정이 기본이어야 하며, 공유 링크 생성 방지 필요 |
| **[MINOR-04]** | **Base64 전송 중 중간자 공격** | MINOR | GAS 웹앱이 HTTPS이므로 전송 구간은 안전하나, Base64 인코딩은 암호화가 아님을 명시할 필요 있음 (향후 혼동 방지) |
