## 이슈 목록 (총 11건)

### 🔴 CRITICAL (1건)

#### C-001 [CRITICAL/기능설계] localStorage 5MB 제한 대비 설계 부재

**문제**: 설계서에서 `localStorage`를 1차 저장소로 채택했으나, **5MB 용량 제한에 대한 방어 로직이 전혀 설계되어 있지 않음**.

**위험 시나리오**:
1. 주디 AI 응답은 마크다운 포맷 + 업무 리스트로 **1건당 평균 2~5KB**
2. 세션당 10턴 대화 가정 시 약 30~50KB
3. 30개 세션 = **최대 1.5MB** 수준이지만...
4. **AI가 긴 업무 브리핑을 반환하는 경우 1회 응답이 10KB+** 될 수 있음
5. 기존 `judy_userName` 등 다른 localStorage 사용과 누적 시 **5MB 초과 → `QuotaExceededError` 발생**
6. 이 에러 미핸들링 시 **전체 채팅 기능 마비** (새 메시지 저장 불가 + 에러 노출)

**개선 방안**:
- `try-catch`로 `setItem` 감싸기 필수
- 용량 초과 시 가장 오래된 세션 자동 삭제하는 **LRU(Least Recently Used) 퇴거 정책** 명세 추가
- 저장 전 `JSON.stringify(data).length` 로 사전 용량 체크 로직 명세 필요
- `maxSessions: 30`은 고정 건수가 아닌 **용량 기반 제한**으로 전환 고려

**심각도 근거**: 방어 로직 없이 배포 시 실서비스에서 `QuotaExceededError`로 채팅 기능 전체 마비 가능 → **서비스 불가** 수준

### 🟠 MAJOR (4건)

#### M-001 [MAJOR/기능설계] 사용자 구분 없는 localStorage 키 설계

**문제**: 설계서의 localStorage 키가 `judy_chat_sessions`로 **사용자 구분이 없음**.

```javascript
// 현재 설계
localStorage.setItem('judy_chat_sessions', JSON.stringify(data));
```

**위험 시나리오**: 공용 PC에서 A유저 로그인 → 채팅 → 로그아웃 → B유저 로그인 시, **B유저가 A유저의 대화 기록을 볼 수 있음**.

**개선 방안**:
```javascript
// 사용자별 키 분리
var storageKey = 'judy_chat_sessions_' + userName;
```

현재 코드에서 `localStorage.getItem('judy_userName')`으로 사용자명을 이미 관리하고 있으므로, 이를 키에 포함해야 함.

---

#### M-002 [MAJOR/보안] XSS 공격 벡터 — 복원 시 innerHTML 주입 위험

**문제**: 설계서에 메시지 복원 시 `innerHTML` vs `textContent` 처리 방법이 명시되지 않음.

현재 코드:
```javascript
// AI 메시지: innerHTML (마크다운 파싱 후)
div.innerHTML = sanitizeHtml(parseMd(content));
// User 메시지: textContent (안전)
div.textContent = content;
```

**위험**: `localStorage`에서 복원할 때 AI 메시지는 `sanitizeHtml(parseMd(content))`를 거쳐야 하는데, 복원 로직에서 이 파이프라인을 누락하면 **저장된 악성 스크립트가 실행될 수 있음**.

**개선 방안**: 설계서에 복원 시 반드시 기존 `appendJudyMsg()` 함수를 재사용하여 동일한 sanitization 파이프라인을 거치도록 명시 필요.

---

#### M-003 [MAJOR/보안] 세션 ID 생성 — 예측 가능한 패턴

**문제**: 설계서의 세션 ID 형식이 `session_1741697950_abc123`으로, 타임스탬프 + 짧은 랜덤 문자열.

```javascript
"id": "session_1741697950_abc123"
```

localStorage는 동일 origin에서 모든 스크립트가 접근 가능하므로 직접적 보안 이슈는 낮지만, 향후 서버 동기화(2차) 추가 시 **세션 ID 충돌/예측 공격**이 가능.

**개선 방안**: `crypto.randomUUID()` 또는 충분한 엔트로피의 UUID v4 사용 권장. 설계서에 ID 생성 규칙 명시 필요.

---

#### M-004 [MAJOR/UX] 드로어 열린 상태에서 대화 영역 높이 부족 — 모바일 미검증

**문제**: 목업 이미지 기준으로 히스토리 드로어가 열리면 대화 영역이 **50% 이하로 축소**됨. 5개 항목이 표시되면 대화 영역이 매우 좁아짐.

**위험**: 모바일(100% 너비 패널)에서 드로어 5개 항목 + 입력 영역까지 합치면 **대화 메시지가 1~2줄만 보임** → UX 심각 저하.

**개선 방안**:
- 드로어 최대 높이 제한 (예: `max-height: 200px; overflow-y: auto`)
- 모바일에서는 드로어를 **오버레이 방식**으로 전환 (대화 영역 위에 겹침)
- 항목 수가 많을 때 스크롤 가능하도록 설계 명시 필요

### 🟡 MINOR (6건)

#### m-001 [MINOR/기능] 대화 제목 "첫 20자" — 한글 잘림 문제

**문제**: "첫 사용자 메시지를 20자로 잘라 제목화"에서, 한글은 20자가 충분하지만 **메시지가 "오늘 마감인 업무와 내일 시작예정인 업무를 알려줘"처럼 길면 잘리는 위치가 어색**할 수 있음.

**개선 권고**: 단어 단위 잘라내기 + "..." 추가 명시 권장.

---

#### m-002 [MINOR/기능] 빈 세션(메시지 0건) 처리 미정의

**문제**: "새 대화" 버튼을 누르고 아무것도 입력하지 않은 채 다른 대화로 전환하면, 빈 세션이 남음. 빈 세션 처리 정책(자동 삭제? 유지?)이 정의되지 않음.

**개선 권고**: 메시지 1건 이상일 때만 세션 저장 정책 추가.

---

#### m-003 [MINOR/UX] 대화 삭제 시 확인 다이얼로그 미설계

**문제**: 🗑️ 아이콘 클릭 시 즉시 삭제인지, 확인 팝업 후 삭제인지 명시되지 않음. 토큰을 소모한 중요 대화를 **실수로 삭제**할 위험.

**개선 권고**: "이 대화를 삭제하시겠습니까?" 확인 다이얼로그 또는 **Undo 토스트**(3초 이내 취소 가능) 설계 추가.

---

#### m-004 [MINOR/UX] 대화 검색 기능 미포함

**문제**: 30개 세션이 쌓이면 원하는 대화를 찾기 어려움. 검색 기능에 대한 언급이 없음.

**개선 권고**: 향후 확장 로드맵에 "대화 제목/내용 검색" 추가 고려.

---

#### m-005 [MINOR/UX] 날짜 그룹핑 미정의

**문제**: 히스토리 목록에서 "오늘", "어제", "지난 주" 등 **시간 기반 그룹핑**이 없으므로 날짜/시간만으로 대화를 구분해야 함.

**개선 권고**: Gemini/ChatGPT처럼 "오늘 / 어제 / 지난 7일" 그룹 헤더 추가 고려.

---

#### m-006 [MINOR/기능] 출근 브리핑 등 특수 키워드 대화의 세션 분리 정책 미정의

**문제**: 현재 코드에는 `'출근'` 키워드 감지 → `handleCheckIn()` 전용 핸들러가 있음. 이런 특수 대화가 일반 대화와 같은 세션에 섞여야 하는지, 별도 세션으로 분리되어야 하는지 정책이 없음.

**개선 권고**: 출근 브리핑은 일반 대화 세션에 포함하되, 설계서에 명시적으로 정책 기술 필요.
