## 1. 배경 및 대표 요구사항

### 1.1 현재 운영 환경

- 5개 에이전트 팀 운영: 자비스, 김감사, 강철, 벙커, 꼼꼼이
- 작업 환경: Claude Code / Antigravity (터미널 기반 AI 에이전트)
- 업무 지시 방식: 자연어 프롬프트 → 각 팀 설계서 읽기 → 에이전트 활성화

### 1.2 대표가 느끼는 문제점

| # | 문제점 | 상세 |
|---|--------|------|
| 1 | **수동 컨텍스트 주입** | 매번 "이 문서 읽고 ○○팀장으로 활동해줘"를 반복 입력 |
| 2 | **직관성 부족** | MD 문서만 보고 승인하는 구조, 터미널 기반이라 시각적 피드백 없음 |
| 3 | **업무 흐름 비가시성** | 5개 팀의 작업 진행 상태가 한눈에 보이지 않음 |
| 4 | **반복 붙여넣기** | 팀 간 핸드오프 시 대표가 직접 결과를 복사해서 다음 팀에 전달 |

### 1.3 대표 비전

> "하나의 대시보드에서 n8n 같은 컨셉으로 내 요청사항이 각 팀에서 어떻게
> 업무가 진행되는지 시각화되고, MD 문서를 보고 승인해주는 시스템"

## 2. 요구 기능 및 난이도 분석

| 기능 | 설명 | 난이도 | 예상 공수 |
|------|------|--------|----------|
| 자연어 입력 → 팀 라우팅 | 대표 지시를 분석해 적절한 팀에 자동 배분 | 🔴 높음 | AI 오케스트레이션 레이어 |
| 업무 플로우 시각화 | n8n식 노드 기반 워크플로우 시각화 | 🔴 높음 | 실시간 상태 동기화, 웹소켓 |
| MD 문서 렌더링 + 승인 UI | MD 파일을 브라우저에서 렌더링, 승인/반려 버튼 | 🟡 중간 | MD 파서 + 승인 워크플로우 |
| 5개 팀 상태 대시보드 | 팀별 현재 작업, 진행률, 핑퐁 상태 표시 | 🟡 중간 | 상태 관리 + API |
| 인증/권한 관리 | 대표/에이전트 구분, 승인 권한 | 🟡 중간 | Auth 레이어 |
| task.md 자동 파싱 | Git의 task.md를 자동 읽어서 진행률 표시 | 🟢 낮음 | 파일 파싱 + 주기적 동기화 |

## 3. 김감사 자문 — 냉철한 분석

### 3.1 문제 진단은 정확함 ✅

대표의 고통 포인트는 실재하며, 장기적으로 대시보드는 필요한 방향임.

### 3.2 우려 사항 3가지 ⚠️

**① 프로세스 미안정화**

- COMMON_RULES.md: 3/1 v1.0 → 3/2 v1.6 (이틀간 6회 변경)
- 팀 설계서: 전면 재설계(v2.0) 직후
- 실전 태스크 처리 이력 부족
- **위험**: 프로세스가 바뀔 때마다 UI도 재작업 필요

> 아직 굳지 않은 콘크리트 위에 건물을 올리는 것과 같다.

**② n8n급 대시보드 = 풀 제품급 공수**

자비스팀이 전담해도 2~4주 소요 예상. 그 기간 동안 본업(서비스 개발) 중단 리스크.

**③ 본질적 병목은 UI가 아님**

현재 병목의 근본 원인:
- 에이전트 세션이 **스테이트리스(stateless)** → 매번 컨텍스트 재주입
- 팀 간 **자동 핸드오프 불가** → 대표가 수동 중계

대시보드를 만들어도 이 두 문제는 해결되지 않음.

### 3.3 QA적 ROI 판정

| 항목 | 판정 |
|------|------|
| 문제 인식 | ✅ 정확함 |
| 해결 방향 (대시보드) | ⚠️ 방향 맞음, **타이밍이 이름** |
| ROI (투자 대비 효과) | ❌ 지금 만들면 낮음 — 프로세스 변경 시 재작업 비용 |
| 권장 시점 | 실전 태스크 5건+ 처리 완료 후 |

## 4. 단계별 구축 권장안

### 4.1 🥇 Phase 0 — 즉시 (0원, 1시간)

**기존 도구를 활용한 즉시 개선**

| 시행 항목 | 도구 | 효과 |
|----------|------|------|
| task를 GitHub Issue로 전환 | GitHub Projects | 칸반 보드에서 상태 추적, 브라우저 승인 |
| 에이전트 호출 워크플로우 | `.agent/workflows/*.md` | 컨텍스트 자동 로딩, 붙여넣기 제거 |
| 팀 간 핸드오프 Issue 템플릿 | GitHub Issue Template | 위임 안내를 Issue로 자동 생성 |

### 4.2 🥈 Phase 1 — 1주 후 (프로세스 안정화 후)

**경량 대시보드 (HTML 1장)**

- task.md 파일들을 파싱해서 팀별 현황판만 표시
- GAS Web App + Google Sheets 조합 → 대표 기존 인프라 활용
- 또는 단순 정적 HTML + GitHub Pages

### 4.3 🥉 Phase 2 — 1개월 후 (필요성 확인 후)

**본격 대시보드 구축**

이 시점에서 프로세스가 안정화되어있으므로:
- 자연어 입력 → 팀 라우팅
- 노드 기반 워크플로우 시각화
- MD 렌더링 + 승인 UI
- 실시간 상태 동기화

기술 스택 후보:
- Frontend: Next.js or Vite + React
- Backend: GAS Web App or Node.js
- 상태 관리: Google Sheets → DB 전환 검토
- 대안: n8n 자체를 도입하여 워크플로우 관리 (커스텀 개발 최소화)
