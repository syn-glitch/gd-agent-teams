# 📝 SYN 회고 분석 및 관리 스킬 (syn-retro-analyzer)

## 1. 언제 사용하나?
- 대표님이 꼼꼼이팀장을 호출하며 "회고록 작성이야"라고 지시할 때.
- 채팅창에 대표님의 일일/주간/월간 회고 원문이 입력되었을 때.

## 2. 핵심 분석 프로토콜 (The 3-View Analysis)
에이전트는 원문을 다음 3가지 관점으로 철저히 분석하여 보고해야 합니다.

1.  **🛡️ 강화점 (Strengths)**: 현재 잘하고 있는 부분, 시스템적으로 안착된 성과, 계속 유지해야 할 긍정적 지표.
2.  **🛠️ 개선점 (Areas for Improvement)**: 현재 병목이 되는 부분, 기술적/심리적 허들, 보완이 필요한 프로세스.
3.  **💡 차기 전략 (Next Step Strategy)**: 분석 결과를 바탕으로 한 구체적인 다음 업무 제언 (자비스/강철 팀 등 타 팀 위임 포함).

## 3. 파일 생성 및 저장 규격
분석이 완료되고 승인되면 다음 규격으로 파일을 생성합니다.

- **저장 경로**: `GD_Agent_teams/SYN_회고/[프로젝트명]/` (예: `Alive/`)
- **파일명**: `YYYY-MM-DD_retrospective.md`
- **필수 포함 섹션**:
    - **Header**: 날짜, 시간, 작성자(SYN), 키워드.
    - **Original Text**: 대표님이 작성한 회고 원문 **(토씨 하나 바꾸지 않고 전체 인용 필수)**.
    - **Analysis**: 꼼꼼이의 3대 관점 분석 내용.
    - **Closing**: 꼼꼼이의 총평 및 응원 메시지.

## 4. 인덱스 업데이트 (INDEX.md)
파일 생성 후 반드시 `GD_Agent_teams/kkoomkkoom-docs/manuals/INDEX.md`는 아니더라도, 회고록 자체 인덱스가 필요할 경우 `SYN_회고/README.md` 등을 업데이트하여 리스트를 관리한다. (기본적으로 꼼꼼이의 아키비스트가 수행)

## 5. 실행 프로세스
1. [이해보고서] 단계에서 3대 관점 분석 결과 및 파일 생성 계획 보고.
2. 대표님 피드백 수렴.
3. 승인 시 파일 생성 (`write_file`).
4. 깃허브 업로드 (`git push`).
5. 최종 링크 보고.

---
**주의사항**: 대표님의 원문은 어떠한 경우에도 요약하거나 생략하지 말고 **원문 그대로** 보존해야 합니다. 분석은 원문 하단에 별도로 추가합니다.
