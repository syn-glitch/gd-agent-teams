# [스타일 가이드] 에이전트 자동화 하네스 표준 (Automation Harness)

---
- **코드명**: `LIGHTWEIGHT_HARNESS_V1`
- **적용 대상**: GitHub Actions, CI 자동화 환경에서 실행되는 모든 에이전트
- **핵심 목표**: 자동화 효율성(Fast-Track), 토큰 절약(Tokenomics), 실행 가시성 확보
---

## 1. 개요: 왜 전용 하네스가 필요한가?

로컬 터미널이나 챗 앱에서 사용되는 **"대화형(Conversational)"** 프롬프트는 사람과의 소통을 위해 긴 설명과 단계별 보고를 포함합니다. 하지만 **"자동화형(Automated)"** 환경에서는 사람이 없으므로 이러한 거치레는 불필요한 비용과 시간을 소모합니다.

**하네스 2.0(Harness 2.0)**은 자동화 목적에 맞게 에이전트의 사고 과정을 극도로 압축하고, 실행 위주로 재구성한 표준입니다.

---

## 2. 하네스 설계 3대 원칙 (L-E-T)

### 2.1 [L] Lightweight (경량화)
- **ALC-Guardrail 절차 생략**: 이해보고서, task.md 생성 및 승인 대기 절차를 제거하고 즉시 도구를 실행하도록 설정합니다.
- **보고 간소화**: "【원본 지시】", "【수행 결과】" 등 정규 보고 형식을 생략하고, 최종 주석(Comment)이나 결과값 위주로 반환합니다.

### 2.2 [E] Efficiency (효율성)
- **컨텍스트 분리**: 에이전트가 읽어야 할 `DESIGN.md` 등 프로젝트 전체 문서를 그대로 넘기지 않고, **작업에 필요한 핵심 코드 레퍼런스**만 추출하여 별도 컨텍스트로 제공합니다.
- **최소 턴(Turn) 지향**: `Read → Edit` 과정을 1-2턴 내에 끝내도록 지시하며, 불필요한 파일 탐색을 금지합니다.

### 2.3 [T] Target-Oriented (목표 지향)
- **목표 명시**: 자동화의 최종 상태(Success State)를 명확히 정의합니다. (예: "이슈 라벨을 'qa-reviewed'로 변경하고 종료")
- **참조 문서 분기**: 대화용 `CLAUDE.md` 대신 자동화용 `AUTOMATION_PROMPT.md` 등을 명시적으로 참조하도록 워크플로우를 설계합니다.

---

## 3. 정량적 제어 가이드 (Control Panel)

| 설정 항목 | 권장값 (CI용) | 설명 |
|:---|:---|:---|
| **Max Turns** | 5 ~ 15 turns | 분석(1~3단계)은 5턴 이내, 코드 수정(4단계)은 15턴 권장 |
| **Max Tokens** | 2,000 ~ 4,000 | CI용 모델(Sonnet/Haiku)의 출력 토큰을 제한하여 비용 절감 |
| **Timeout** | 5 ~ 10 mins | 자급 자족형 코드 수정 시 최대 10분, 단순 분석은 2분 이내 |

---

## 4. 자동화 프롬프트 모범 사례 (Best Practice)

### [예시] 자비스 자동 코드 수정 프롬프트
```markdown
## CI 자동화 환경 규칙
1. ALC-Guardrail(이해보고서/승인/task.md) 절차를 수행하지 않는다.
2. 역할 경계 라우팅(위임 안내)을 수행하지 않는다.
3. 즉시 Read 도구를 사용하여 대상 파일을 확인한다.
4. 분석 내용을 요약하지 말고 즉시 Edit으로 코드를 수정한다.
5. 수정 완료 후 성공 여부(true/false)만 반환한다.
```
