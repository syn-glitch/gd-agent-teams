## 1. 🤝 협업 프로세스 실전 사례 (ALC-Guardrail)

대표님과 저희 팀이 실제로 채팅을 통해 협업하는 과정을 단계를 나누어 재구성했습니다.

### [사례: "지출 관리 대시보드 기획해줘"]

#### 📌 STEP 1. 이해 보고 (Alignment)
- **대표님**: "벙커 팀, 이번에 지출 관리 대시보드 기획 좀 해줘."
- **에이전트(벙커)**: "이해했습니다! 지침 확인 결과, 기존 지출 내역을 시각화하고 AI 회고를 포함한 대시보드 PRD를 작성하는 건이군요. 작업 범위는 기획안 작성까지이며, 배포는 포함되지 않습니다. **승인해 주시면 [계획] 단계로 진입하겠습니다.**"

#### 📝 STEP 2. 계획 보고 (Task Planning)
- **대표님**: "그래, 진행해."
- **에이전트(벙커)**: (내부적으로 `task.md` 생성 후) "계획 보고드립니다. 1단계로 요구사항 상세 분석, 2단계로 UI 와이어프레임 설계, 3단계로 최종 PRD 문서를 생성하겠습니다. 총 예상 시간은 15분입니다. **승인해 주시면 실행하겠습니다.**"

#### 📋 STEP 3. 완료 보고 (Completion)
- **대표님**: "ㅇㅇ"
- **에이전트(벙커)**: (작업 수행 후) "업무 완료 보고드립니다. `PRD_Dashboard.md` 작성을 마쳤습니다. 
  - 【수행 결과】 요구사항 5건 반영 완료.
  - 【산출물】 `GD_Agent_teams/bunker/tasks/2026-03/PRD_Dashboard.md`
  - 【위임 사항】 UI 구현이 필요하므로 **자비스 개발팀**에 지시 부탁드립니다.
  - 【토큰 사용량】 12,500 토큰 ($0.12)"

## 2. 🔀 역할 경계 및 위임 사례 (Role Boundary)

에이전트가 완벽하지 않기에, 자신의 역할을 넘어서는 실수를 방지하는 핵심 장치입니다.

### [실제 사례: QA 중 버그 발견 시]
- **상황**: **김감사 QA팀**이 자비스가 만든 코드를 검수하다가 오타를 발견함.
- **나쁜 예 (경계 침범)**: 김감사 에이전트가 "오타를 발견해서 제가 직접 수정했습니다"라고 보고함. → *문제가 생겼을 때 누가 수정했는지 추적이 안 됨.*
- **좋은 예 (위임)**: 김감사 에이전트가 "QA 결과, `Code.gs` 12번 라인에 오타를 발견했습니다. **고치는 것은 자비스 팀의 역할이므로**, 아래 위임 안내를 참고하여 자비스 팀에 수정을 지시해 주세요."
    - **결과**: 대표님은 김감사의 보고를 보고 자비스에게 "김감사가 찾은 버그 수정해"라고 한 줄만 지시하면 됩니다. **책임 소재와 이력이 명확해집니다.**

## 4. 🩸 고통 리포트 (Pain Report) — 실패에서 배운 교훈

가장 뼈아픈 실패 사례이자, 현재의 강력한 룰이 만들어진 배경입니다.

### 💀 [2026-03-03 대시보드 개발 실패 사건]
- **사건 개요**: 새로운 대시보드 프로젝트를 기존 프로젝트 폴더에서 분리하지 않고 진행하다가 코드가 엉키고 배포가 수차례 실패함.
- **결과**: **10시간 이상의 작업물 전량 삭제** 및 대표님의 코드 삭제 지시.
- **핵심 교훈 (고통 리포트 내용)**:
  1. **"동상이몽의 고통"**: 기획서(PRD)가 확정되지 않았는데 개발을 시작하면 결국 산출물을 버리게 됨.
  2. **"환경의 고통"**: 새 프로젝트는 반드시 새 폴더, 새 세션에서 시작할 것.
  3. **"오보고의 고통"**: "성공했습니다"라는 보고를 맹신하지 말고 실제 링크나 결과물을 눈으로 확인할 것.

> **"같은 실수의 반복은 실수가 아니라 형편없는 실력과 팀이다."** 
> — 이 문장은 현재 모든 에이전트의 시스템 프롬프트에 깊이 새겨져 업무의 무게중심을 잡아주고 있습니다.

## 🚀 5. 향후 개발 방향 및 내부 개발자 피드백

용남 대표님께서 제시하신 **GD Agent Teams의 중장기 비전 및 내부 개발자들을 위한 가이드라인**입니다.

### 🎯 전사 에이전트 팀 구축 계획 (1개월 이내)
- 현재 프로젝트 단위로 활동하는 각 팀을 **전사 차원의 공식 에이전트 조직**으로 격상하여 운영을 내재화합니다.

### 🛠️ 기술적 진화 및 서버 배포 전략
- **GAS 한계 극복**: 현재 Google Apps Script(GAS) 기반 인프라의 로딩 지연(Slack/Web) 문제를 해결하기 위해, 현재 코드를 기반으로 한 **독립 서버 배포**를 추진합니다.
- **개발 방식 확인**: 현재 우리가 사용하는 방식은 **클로드 코드(Claude Code)에서 제공하는 에이전틱 워크플로우**가 맞으며, 이를 더욱 고도화하여 프로덕션 레벨로 끌어올리는 단계입니다.

### 🤝 지식 공유 및 데이터 자산화 (DB화)
- **스킬(Skills) 공유**: 각 팀원이 업무 중 생성한 유용한 스킬들을 전사 저장소에 공유하여 상호 역량을 강화합니다.
- **마크다운(Markdown) 중심 DB 구축**: 
    - 슬랙 대화, 개인 메모, 회의록 등 회사의 모든 텍스트 기반 기록을 **마크다운 형식**으로 통일합니다.
    - 이를 데이터베이스(DB)화하여 나중에 에이전트들이 언제든 검색하고 학습할 수 있는 **사내 지식 베이스**로 활용합니다.
